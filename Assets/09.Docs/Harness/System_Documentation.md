**작성일**: 2026-04-11
**버전**: v1.0

## 1. 하네스란?

하네스는 프롬프트 엔지니어링 기법으로, **AI 에이전트를 구조화된 틀 안에서 제어**하는 방법론입니다.
자유롭게 두는 대신, 프로젝트 컨텍스트·코딩 규칙·제약 조건·작업 흐름을 체계적으로 조합하여
AI가 **안전하고 일관되며 예측 가능한 결과**를 내도록 유도합니다.

### 핵심 원칙

```
자율성과 제어의 균형
  → AI가 창의적으로 작업하되, 프로젝트 기준에서 벗어나지 않도록 제약
```

## 4. 각 하네스 구성 요소 상세

### 4.1 CLAUDE.md — 프로젝트 컨텍스트

AI가 프로젝트를 이해하는 데 필요한 모든 배경 정보를 제공합니다.

**포함 내용:**
- 폴더 구조 전체 맵핑
- 핵심 클래스 관계도 (BattleSceneSetup → TurnManager → SkillDrawSystem → ...)
- Character 컴포넌트 합성 구조 (Health, Stats, StatusEffects, SkillInventory)
- 턴 사이클 정의 (`Draw → PlayerAction → Execution → EnemyTurn → BattleEnd`)
- 데이터 계층 (CharacterData, SkillData ScriptableObject)
- 개발 Phase 현황 (Phase 1-2 완료, Phase 3-4 미착수)
- 기술 스택 (Unity 6, URP, TextMesh Pro, New Input System)

**목적:** AI가 "프로젝트를 이미 알고 있는 팀원"처럼 작동할 수 있게 함.

### 4.2 CLAUDE.md — 코딩 규칙

AI가 생성하는 코드의 일관성을 보장합니다.

| 규칙 | 내용 |
|------|------|
| 네임스페이스 | `TeamLog` 최상위, 폴더 구조에 따른 하위 네임스페이스 8종 정의 |
| 이벤트 기반 통신 | 클래스 간 직접 참조 최소화, C# event/Action 사용 |
| UI-로직 분리 | UI는 표시만, 게임 로직은 Combat/Characters 계층에 |
| 순수 C# 우선 | MonoBehaviour는 Unity 라이프사이클이 필요한 경우만 |
| ScriptableObject | 모든 게임 데이터를 ScriptableObject로 관리, 하드코딩 금지 |
| 네이밍 | PascalCase(클래스), camelCase(필드), On접두사(이벤트), Class_Name(SO) |
| 파일 배치 | 시스템별 해당 폴더, UI는 항상 `UI/Battle/`, Editor는 항상 `Editor/` |

### 4.3 CLAUDE.md — 가드레일

AI가 절대 하지 말아야 할 행위를 명시합니다.

1. `Library/`, `Temp/`, `obj/` 폴더 수정 절대 금지
2. `.meta` 파일 수동 생성/수정 금지
3. `FindObjectOfType`, `Find`, `GameObject.Find` 사용 금지
4. `PlayerPrefs` 사용 금지
5. 불필요한 MonoBehaviour 상속 금지
6. `02.Scripts/` 외부에 게임 스크립트 배치 금지
7. UI 스크립트에서 게임 로직 구현 금지
8. 기존 public API 변경 시 하위 호환성 확인

### 4.4 CLAUDE.md — 워크플로우

새 기능 추가 시 AI가 반드시 따라야 하는 순서:

```
1. 기존 코드 읽기   → 관련 클래스 먼저 파악
2. 데이터 정의      → ScriptableObject부터 설계
3. 로직 구현       → 순수 C# 클래스로 핵심 작성
4. 이벤트 연결      → BattleEventManager 또는 클래스 이벤트로 UI 연동
5. UI 구현         → UI/Battle/에 컴포넌트 작성
6. 에디터 도구      → DataGenerator, SceneBuilder 업데이트
7. 테스트          → 씬에서 검증
```

### 4.5 CLAUDE.md — 가비지 컬렉터

프로젝트 오염을 정기적으로 감지하고 제거하는 시스템.

**수집 대상 7가지:**
| 카테고리 | 감지 기준 |
|----------|-----------|
| 죽은 코드 | 참조되지 않는 메서드, 클래스, 변수 |
| 미사용 에셋 | 씬/스크립트에서 참조되지 않는 SO, 프리팹 |
| 기획 불일치 | GameDesign.md와 충돌하는 구현 |
| 아키텍처 위반 | CLAUDE.md 규칙을 위반한 코드 |
| 중복 로직 | 동일 로직이 여러 위치에 산재 |
| 유령 참조 | 존재하지 않는 대상 참조 |
| 스텁/TODO 방치 | 미구현 상태로 장기 방치 |

**수집 주기:**
- 경량 수집: 매 세션 종료 전 (수정 파일만)
- 심층 수집: Phase 전환 시 또는 사용자 요청 (전체 프로젝트)

**수집 절차:**
```
스캔 → 분류(즉시삭제/확인필요/보류) → 보고 → 정화(사용자 승인 후) → 검증
```

## 5. 프롬프트 템플릿별 프로세스 상세

### 5.1 버그 수정 (bug-fix.md)

```
입력: 버그 설명 → 재현 단계 → 기대 동작 → 실제 동작 → 관련 파일

Step 1: 원인 파악
  - 관련 코드 읽기, Unity 콘솔 확인, 씬 상태 확인, git diff

Step 2: 수정
  - 최소 변경, API 시그니처 유지, 연쇄 영향 분석

Step 3: 검증
  - 에러 확인, 엣지 케이스 고려 (HP 0, 빈 타겟, 중복 상태이상)
```

### 5.2 기능 추가 (feature-add.md)

```
입력: 기능명 → 설명 → 관련 시스템 → 디자인 문서 참조

Step 1: 기존 코드 분석 (패턴/컨벤션/이벤트 구조 파악)
Step 2: 데이터 계층 설계 (ScriptableObject, enum 확장)
Step 3: 로직 구현 (순수 C#, 이벤트 정의)
Step 4: UI 구현 (다크 테마, NanumGothic, 타입 색상 규칙)
Step 5: 에디터 도구 업데이트
Step 6: 검증 (컴파일, 회귀 테스트)

시스템별 참고 패턴 포함:
- 새 상태이상: enum 추가 → EffectProcessor → SkillData → UI
- 새 적 행동: 액션 타입 → 의도 표시 → TurnManager
- 새 캐릭터: enum 확장 → 스킬 4개 → DataGenerator → UI 색상
```

### 5.3 리팩토링 (refactor.md)

```
입력: 대상 → 목적(성능/가독성/중복제거/확장성) → 상세 설명

Step 1: 영향 범위 분석 (참조, API, 구독자, 호환성)
Step 2: 원칙 수립 (동작변경 없음, 점진적 변경)
Step 3: 패턴 적용
  - 컴포넌트 분리 (MonoBehaviour → 순수 C# + 래퍼)
  - 이벤트 도입 (직접 호출 → 이벤트 구독)
  - 매직넘버 제거
Step 4: 검증 (컴파일, 씬 동작, 이벤트 연결, SO 참조)

주의사항: BattleUISceneBuilder(1700줄), EffectProcessor(static),
         BattleEventManager(싱글톤)
```

### 5.4 데이터 설계 (data-design.md)

```
입력: 콘텐츠 타입 → 컨셉 → 밸런스 기준

Step 1: 기존 데이터 분석 (스탯 범위, 위력 범위)
Step 2: 설계 원칙 적용
  - 캐릭터: HP 50-150, ATK 5-20, DEF 2-10
  - 스킬: Weight 1-10, Power 역할별, Cost 0
  - 데미지 공식: max(1, ATK + Power - DEF)
Step 3: 구현 체크리스트 (캐릭터/스킬/적별)
Step 4: 밸런스 검증
```

### 5.5 가비지 컬렉터 (garbage-collect.md)

```
입력: 수집 타입(경량/심층) → 범위

경량 수집 (세션 종료 전):
  죽은 코드 스캔 → 일관성 스캔 → 즉시 정화

심층 수집 (Phase 전환 시):
  Phase 1: 스캔 (7가지 오염 유형 전체 탐지)
  Phase 2: 분류 및 보고 (보고서 양식에 맞춰 정리)
  Phase 3: 정화 (사용자 승인 후 삭제, 삭제 순서: 데이터→스크립트→씬)
  Phase 4: 기록 (삭제 항목 수, 보류 수, 다음 권장 시점)

오염 방지 체크리스트:
  기획 일치 / 규칙 준수 / 중복 없음 / 사용처 명확 / TODO 구체화
```

## 8. 사용 방법 (How to Use)

### 8.1 기본 사용 — Claude Code 실행

Claude Code를 실행하면 **CLAUDE.md가 자동으로 로드**됩니다.
별도 설정 없이 AI가 프로젝트 컨텍스트, 규칙, 가드레일을 인지한 상태로 작업을 시작합니다.

```bash
# 프로젝트 폴더에서 Claude Code 실행
cd C:\Unity\Projects\Team_Log
claude
```

### 8.2 작업별 사용법

#### 버그 수정
```
# Claude Code에게 이렇게 요청:
"버그 수정해줘"
"데미지 계산이 이상해. 적이 방어력을 무시하고 있어"

# AI가 bug-fix.md 프로세스를 따라 작동:
# 1. 관련 코드 읽기 → 2. 원인 파악 → 3. 최소 수정 → 4. 검증
```

**입력 예시:**
```
### 버그 설명
적 공격이 방어력을 무시함

### 재현 단계
1. BattleUI 씬 실행
2. 적 턴에 적이 공격
3. 전사(DEF 8)가 22 데미지를 받음 (기대: 14)

### 기대 동작
ATK + damage - DEF 공식이 적용되어야 함

### 실제 동작
ATK + damage가 그대로 HP에 반영됨

### 관련 파일
Assets/02.Scripts/Combat/AI/EnemyAIController.cs
```

#### 기능 추가
```
# Claude Code에게 이렇게 요청:
"새 상태이상 '빙결' 추가해줘"
"적 보스 시스템 만들어줘"

# AI가 feature-add.md 프로세스를 따라 작동:
# 1. 기존 코드 분석 → 2. 데이터 설계 → 3. 로직 구현 →
# 4. 이벤트 연결 → 5. UI 구현 → 6. 에디터 도구 → 7. 검증
```

**입력 예시:**
```
### 기능명
보스 몬스터 시스템

### 설명
다중 페이즈 보스. HP 50% 이하에서 패턴 변경.
페이즈 전환 시 버프 획득.

### 관련 시스템
Combat/AI, Combat/Turn, UI/Battle

### 디자인 문서 참조
GameDesign.md 섹션 4.2 — 적 타입 (보스: 다중 페이즈, 특수 메커니즘)
```

#### 리팩토링
```
# Claude Code에게 이렇게 요청:
"BattleSceneSetup 리팩토링해줘"
"중복된 HP 바 로직 통합해줘"

# AI가 refactor.md 프로세스를 따라 작동:
# 1. 영향 범위 분석 → 2. 원칙 수립 → 3. 패턴 적용 → 4. 검증
```

### 9.3 refactor.md (리팩토링)

**목적:** 기존 코드를 개선하면서 동작은 유지

**입력 양식:**
```
### 리팩토링 대상
[파일 경로 또는 시스템 이름]

### 목적
[성능 개선 / 가독성 향상 / 중복 제거 / 확장성 확보 중 선택]

### 상세 설명
[무엇을 어떻게 개선하고 싶은지]
```

**AI 작업 프로세스:**

Step 1: 영향 범위 분석
1. 리팩토링 대상 파일의 모든 참조 검색
2. public API 변경 여부 확인
3. 이벤트 구독자 파악
4. 하위 호환성 영향 평가

Step 2: 리팩토링 원칙
- **동작 변경 없음**: 외부 관찰 가능한 동작 동일 유지
- **점진적 변경**: 한 번에 하나의 관심사만
- **테스트 보장**: 수정 전후 동일 시나리오 검증
- **기존 패턴 준수**: 컨벤션에서 벗어나지 않음

Step 3: 일반적인 리팩토링 패턴

컴포넌트 분리:
```
Before: 하나의 큰 MonoBehaviour
After: 순수 C# 로직 클래스 + MonoBehaviour 래퍼
예시: Character (순수 C#) + CharacterData (SO) + CharacterUIPanel (MB)
```

이벤트 도입:
```
Before: A가 B를 직접 호출
After: A가 이벤트 발생 → B가 구독
public event Action<int> OnHPChanged;
public event Action OnDeath;
```

매직넘버 제거:
```
Before: if (hp <= 0)
After:  if (healthComponent.IsDead)
Before: damage = atk + 15 - def;
After:  damage = statComponent.ATK + skill.Power - target.DEF;
```

Step 4: 검증
1. 컴파일 에러 없는지 확인
2. 기존 씬에서 정상 동작 확인
3. 이벤트 연결 끊어지지 않았는지 확인
4. ScriptableObject 참조 유효한지 확인

**주의사항:**
- BattleUISceneBuilder (~1700줄): 프로그래박틱 UI 구성, 구조 변경이 씬 전체에 영향
- EffectProcessor: static 클래스, 확장 시 인스턴스화 필요성 검토
- BattleEventManager: 싱글톤, 다른 싱글톤과의 의존성 주의
